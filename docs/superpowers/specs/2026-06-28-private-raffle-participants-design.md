# Private Raffle Participants Design

## Goal

Показывать победителя и участников завершённого розыгрыша по игровым никнеймам, не публикуя идентификаторы, которые могут упростить деанонимизацию.

## Scope

Изменение затрагивает два репозитория:

- `squadjs2` формирует безопасный публичный raffle snapshot;
- `autoseed` нормализует этот контракт и показывает список участников в истории победителей.

Внутреннее SQL-хранилище, выбор победителя и личный `warn` продолжают использовать EOS/Steam ID. Приватность применяется только на границе публичного snapshot и в DOM сайта.

## Approaches

### 1. Раскрываемый список в каждой записи истории

Запись розыгрыша показывает победителя, приз, дату и количество участников. Элемент `details` с заголовком `Участники (N)` раскрывает список игровых никнеймов.

Преимущества: история остаётся компактной, решение доступно с клавиатуры без дополнительного JavaScript, хорошо масштабируется на мобильном экране.

### 2. Всегда раскрытый список

Все никнеймы видны сразу. Это проще технически, но при нескольких розыгрышах и десятках участников страница становится слишком длинной и плохо сканируется.

### 3. Отдельное модальное окно

Список открывается поверх страницы. Это сохраняет компактность, но добавляет управление фокусом, закрытие, mobile overlay и отдельный компонент без продуктовой необходимости.

Выбран подход 1.

## Public Data Contract

`Raffle.getSnapshot()` возвращает для `history[].winner`, `history[].participants[]` и `history[].startedBy` только:

```json
{
  "name": "Player Name",
  "joinedAt": "2026-07-14T18:05:00.000Z"
}
```

Поля `steamID`, `eosID` и Discord ID не входят в публичный ответ. Данные в БД не мигрируются и сохраняют идентификаторы для внутренней логики.

Frontend принимает старый payload с дополнительными полями, но `mapRaffleParticipant()` создаёт новый объект только из `name` и `joinedAt`. Так идентификаторы не распространяются дальше слоя нормализации даже во время rolling deploy.

## UI

Каждая завершённая запись содержит:

- игровой никнейм победителя или `без победителя`;
- приз;
- дату, сумму и количество участников;
- раскрываемую строку `Участники (N)`;
- список никнеймов после раскрытия;
- текст `Участников не было`, если список пуст.

Никнеймы переносятся внутри доступной ширины. Steam, EOS и Discord ID не рендерятся ни визуально, ни скрытым текстом.

## Error Handling

- Отсутствующий или пустой `name` нормализуется в `Игрок`.
- Некорректный `participants` нормализуется в пустой массив.
- Отсутствующий победитель не мешает показать участников.
- Старый backend payload с ID остаётся совместимым, но ID отбрасываются.

## Verification

- Backend test подтверждает, что публичная история содержит никнеймы, но не содержит `steamID`/`eosID`, при этом завершение розыгрыша всё ещё использует внутренний `winner.eosID`.
- Frontend E2E раскрывает список, проверяет оба никнейма, точное количество и отсутствие тестовых Steam/EOS/Discord ID в `body`.
- Desktop и mobile QA проверяют раскрытие, длинный никнейм, console health и отсутствие горизонтального overflow.
- После деплоя проверяются public snapshot всех трёх exporter-ов и GitHub Pages `/#winners`.
