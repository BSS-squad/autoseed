# Planned Raffle Notifications Design

## Goal

Отделить будущие серии розыгрышей от уже идущей серии и показать планы как заметные, но ненавязчивые уведомления на странице победителей.

## Behavior

- Кампания считается будущей, если `startsAt` позже текущего времени.
- Каждая будущая кампания выводится над сводкой отдельным уведомлением с текстом «Планируется серия розыгрышей. Не пропустите», датами, прайм-окном, порогом игроков и банком.
- Одновременно может отображаться несколько будущих кампаний.
- Одинаковые кампании из нескольких exporter snapshot-ов дедуплицируются.
- Карточка «Серия» в основной сводке показывается только для кампании, которая уже началась и ещё не закончилась.
- Активные розыгрыши, история и бюджет сохраняют текущую логику.

## Data Contract

Frontend принимает новый `raffles.campaigns` как массив кампаний и продолжает читать legacy-поле `raffles.campaign`. Оба источника объединяются и дедуплицируются, поэтому backend может перейти на массив без одновременного breaking change.

Для текущего production snapshot, где одна июльская кампания повторяется на трёх серверах, страница покажет одно уведомление.

## UI

Уведомления располагаются отдельной полноширинной секцией между hero и основной сводкой. Это не modal: пользователь не должен закрывать окно и не теряет доступ к истории победителей.

Основная сводка остаётся компактной. До начала кампании в ней нет карточки «Серия»; после начала карточка появляется с фактическими параметрами текущей кампании.

## Verification

- E2E fixture содержит две будущие кампании и дубликат одной из них на другом exporter-е.
- Тест проверяет два уведомления, отсутствие карточки текущей серии и сохранение active/history/budget.
- Отдельное состояние с текущей кампанией проверяет появление карточки «Серия».
- Build и полный Playwright suite проходят.
- Desktop и mobile rendered QA проверяют отсутствие overflow, console errors и layout overlap.
